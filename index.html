Реальная топология on-chain (JustLend, Netts, наш supply-узел) в диаграммах. Все цифры сняты с mainnet через TronGrid.

01Сколько энергии и куда течёт

02Кто кому делегирует

Два контура. Слева оптово-делегационный: JustLend стейкает миллиарды, op pool арендует кусок и раздаёт рознице. Справа агрегатор поставок: стейкеры заливают энергию в пулы, пулы жгут её сами за клиентов.

03Откуда энергия, которой быть не должно

Счёт клиента: стейка 0, энергии 0 - а USDT уходит. Главный механизм - JIT-делегация: энергия живёт один блок.

sequenceDiagram
  autonumber
  participant P as Пул-поставщик
  participant C as Клиент
  participant T as Сеть TRON
  Note over C: стейк 0, энергия 0
  P->>C: Блок N: DelegateResource (~7000 TRX веса)
  Note over C: появилось ~65K энергии
  C->>T: Блок N: перевод USDT сжигает 65K
  P->>C: Блок N+1: UnDelegateResource
  Note over C: снова стейк 0, энергия 0
  Note over P: тот же стейк уходит следующему клиенту

Сброс счётчика used

Расход пишется на получателя, не на стейкера. Свежий адрес = used 0 = снова полный лимит.

flowchart LR
  S["1 стейк
7000 TRX"]-->A["адрес A
used 0 → 65K"]
  S-->B["адрес B
used 0 → 65K"]
  S-->D["адрес C
used 0 → 65K"]
  A & B & D -->R["N × 65K за сутки
из одной TRX"]
  classDef a fill:#2a2113,stroke:#7d6320,color:#f5d99a;
  classDef hl fill:#3a2e12,stroke:#f5b342,color:#ffe9b8;
  class S,A,B,D a; class R hl;
Почему сеть это терпит. В каждый момент вес стейка отдан одному адресу, мгновенная ёмкость сети не превышается. Кратность набегает во времени: расход списывается на разные адреса с независимыми счётчиками. Ограничитель - не дневной лимит, а пропускная способность (каждый delegate и undelegate это транзакция + bandwidth).

04На чём маржа

Семь рычагов: часть честная экономика, часть выжимание механики блокчейна.
